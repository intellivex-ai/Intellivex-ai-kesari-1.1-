## 2026-04-15 - [Fix IDOR in api/generate-image.ts]
**Vulnerability:** Insecure Direct Object Reference (IDOR) where any authenticated user could insert image messages into any other user's chat.
**Learning:** The application was missing explicit authorization checks (resource ownership verification) on the server side when performing operations via API endpoints that accept identifiers (like `chatId`) from the client.
**Prevention:** Always verify resource ownership (e.g., query the `chats` table to match the `user_id` with the authenticated `userId`) before allowing modifications or creations on that resource.

## 2024-05-31 - [XSS via iframe srcdoc in Workspace Sandbox]
**Vulnerability:** The code execution sandbox in `src/stores/workspaceStore.ts` dynamically injected user-provided code directly into a `<script>` tag within an iframe's `srcdoc` attribute without serialization: ``sandboxFrame.srcdoc = `<script>...${code}...</script>` ``. This permitted Cross-Site Scripting (XSS) / Sandbox Breakout vulnerabilities because a user could provide a string like `</script><script>alert(1)</script>`, which immediately closed the script tag and started a new one to execute malicious code.
**Learning:** Even within an isolated iframe (with `sandbox="allow-scripts"`), directly interpolating unsanitized code into an HTML context allows HTML injection that can break the structure of the `<script>` tag and potentially compromise the sandbox's intended restrictions (e.g. escaping the `try/catch` wrapper and `postMessage` isolation).
**Prevention:** Always serialize untrusted code when injecting it into an HTML context. Using `eval(${JSON.stringify(code).replace(/</g, '\\u003c')})` securely escapes HTML characters (specifically `<` and `>`) while preserving the code's valid execution semantics, preventing HTML injection/XSS breakouts.
